pandabytes MCP server

The hub ships an MCP (Model Context Protocol) server so AI agents can query the catalog natively — no HTTP plumbing. It wraps the same read-only surface as the REST API: an MCP call never triggers an upstream fetch, it only reads the already-built catalog, so any query volume is safe and quota-free. The tools mirror the REST field names verbatim, and errors surface as the REST error envelope ({ ok:false, error, code, hint }) so an agent can self-correct on hint.

The MCP exposes both the freshness tools (source board, source detail, 30-day coverage, hub stats, gap finder) and the editorial KPI tools (top articles, per-article performance, top authors / categories / teams) — so an agent can ask "which pandabytes sources are stale?", "chart GA4 pageviews for the last 30 days", or "who were our top authors by pageviews this month?" without writing any HTTP code.

Install & configure

Install it with uvx and point it at your key:

uvx mcp-pandabytes

Configure it in your MCP client (e.g. Claude Desktop / Claude Code mcp.json):

{
  "mcpServers": {
    "pandabytes": {
      "command": "uvx",
      "args": ["mcp-pandabytes"],
      "env": {
        "PANDABYTES_API_KEY": "YOUR_KEY",
        "PANDABYTES_API_BASE": "https://pandabytes.restful.zone/v1"
      }
    }
  }
}

Need a key? Request one on the Get a key page — the operator approves by email and your key is emailed to you. list_sources, get_stats and friends are read-only and quota-free once you have it.

The 10 tools

Ten tools in two families — 5 freshness/catalog + 5 editorial KPI — each a thin read-only call to the REST API. Field names mirror the REST contract verbatim. Status enum: FRESH · STALE · STALLED · MISSING · FROZEN · DEPRECATED; kind enum: spine · metric · reference · vector · federated · derived.

Freshness & catalog

toolwhat it doesREST
list_sources(kind?, status?) The freshness board — every source + health at a glance; find STALE/MISSING/STALLED; pick an id. Optional kind/status filters. GET /v1/sources
get_source_status(id) One source in depth: metrics, key, cadence (load), days_behind, earliest/latest date, notes. GET /v1/sources/{id}
get_coverage(id, days=30) Daily coverage timeseries (gap-free, every day present) — reason about dips/ramps/holes over time. days 1..365. GET /v1/sources/{id}/coverage
get_stats() Hub top-line counts — total fact rows, sources total, status tally, biggest tables. "How big is the hub?" GET /v1/stats
find_gaps(id, days=30) Only the HOLES — zero-row days inside a source's active range, not the full series. GET /v1/sources/{id}/coverage

Editorial KPIs

toolwhat it doesREST
get_top_articles(metric="pageviews", days=30, limit=20) Rank articles by a metric over a window. "Best-performing articles this month." GET /v1/articles/top
get_article(page_path) One article + cross-source metrics: metadata + all-time canonical KPIs + per-source breakdown. GET /v1/articles?path=
get_author_performance(days=30, limit=50) Author leaderboard — article count, pageviews, avg engagement time, time-spent. "Which authors drove the most traffic?" GET /v1/perf/author
get_category_performance(days=30, limit=50) Category leaderboard (primary category) — compare verticals: Relationships, Curiosities, Funny, … GET /v1/perf/category
get_team_performance(days=30, limit=50) Team leaderboard, grouped by editorial team_id (gateway category taxonomy). "How did each team perform?" GET /v1/perf/team

Every tool returns a single JSON object; full field-by-field shapes are in the REST API usage and Editorial performance sections of the docs (the MCP tools return the same payloads).
