pandabytes MCP server

The hub ships an MCP (Model Context Protocol) server so AI agents can query the catalog natively — no HTTP plumbing. It wraps the same read-only surface as the REST API: an MCP call never triggers an upstream fetch, it only reads the already-built catalog, so any query volume is safe and quota-free. Errors surface as the REST error envelope ({ ok:false, error, code, hint }) so an agent can self-correct on hint.

The surface is deliberately minimal and universal — exactly three tools: describe_schema() returns the data map (every queryable relation + columns + join recipes + worked examples), query(sql) runs one sandboxed read-only SELECT over the whole catalog, and agentic_query(question) is an escape hatch that lets an inner agent explore over several rounds. Together query + the schema doc subsume every narrow per-topic tool — an agent can ask "which pandabytes sources are stale?", "chart GA4 pageviews for the last 30 days", or "who were our top authors by pageviews this month?" by composing its own SELECT, no HTTP code and no permission-prompt sprawl.

There is one canonical route for every client: the remote URL below. Streamable HTTP, nothing to install — the same single URL works in claude.ai, Claude Code, Claude Desktop, Cursor, and any other MCP client.

Connect (works everywhere)

One remote MCP server, one URL, the same 3 read-only tools in every client. The access token rides in the URL query string (?k=) and is checked at the edge; the hub's own data key stays server-side, so the connector never sees it. Paste this single URL — nothing to install, no uvx:

https://pandabytes.restful.zone/mcp?k=<your-token>

Replace <your-token> with the token the operator gives you — the full URL, token included, is the one thing you paste. The connector is read-only: no raw SQL, no writes; every call just reads the already-built catalog, so any query volume is safe and quota-free. Need a token? Request one on the Get a key page.

First calls (quickstart)

Once connected, the fastest path is two tools. Call describe_schema() first — it returns the queryable relations + their real column names and worked examples — then compose a read-only SELECT and run it with query(sql) (SELECT-only, capped at 1000 rows). Example:

describe_schema()
query("SELECT page_path, title, bp_views FROM v_articles ORDER BY bp_views DESC LIMIT 5")

Column-name gotcha. In SQL the article metrics are bp_views, bp_upvotes, and categories (an array) — not the friendlier pageviews / category you may see in the canned endpoints. Always run describe_schema() first to get the exact column names rather than guessing; a wrong name comes back as an error with a hint you can self-correct on.

Testing the endpoint? This is a Streamable-HTTP MCP server, not a plain JSON-RPC endpoint. A bare POST /mcp?k=<token> with a hand-rolled JSON-RPC body returns [] (empty) — that is expected, not a failure. A real client must do the Streamable-HTTP / SSE session handshake (initialize → session id → tool calls). Use a proper MCP client to connect: Claude Code claude mcp add --transport http (below) or the claude.ai / Claude Desktop Add custom connector dialog — don't curl a raw JSON-RPC POST and conclude it's down.

Per-client setup

Same URL, four ways to add it. Once added, ask Claude things like "which pandabytes sources are stale?" or "top BoredPanda authors by pageviews this month" and it calls these tools directly.

clienthow to add the remote URL
claude.ai (web) Settings → Connectors → Add custom connector. Name Pandabytes, paste the URL as the Remote MCP server URL, leave OAuth Client ID and Client Secret blank.
Claude Code (CLI) One command — pass the full URL in quotes (verified ✔ Connected): 
claude mcp add --transport http pandabytes \
  "https://pandabytes.restful.zone/mcp?k=<your-token>"
Check it with claude mcp list; remove with claude mcp remove pandabytes.
Claude Desktop Add it as a remote connector (Settings → Connectors → Add custom connector), same URL — no uvx, nothing to install.
Cursor / other Any MCP-aware client: add an HTTP / remote server pointed at the same URL. The ?k= token is the only credential.
Optional: local stdio (offline / development)

The remote URL above is the recommended path for all clients. This local uvx stdio mode exists only for offline use or developing against the hub directly — most users should ignore it. It runs the same server on your machine over stdio and talks to the REST API with your own data key (not the ?k= edge token):

uvx mcp-pandabytes

Configure it in your MCP client's mcp.json:

{
  "mcpServers": {
    "pandabytes": {
      "command": "uvx",
      "args": ["mcp-pandabytes"],
      "env": {
        "PANDABYTES_API_KEY": "YOUR_KEY",
        "PANDABYTES_API_BASE": "https://pandabytes.restful.zone/v1"
      }
    }
  }
}

Need a key? Request one on the Get a key page.

The 3 tools

The surface is minimal and universal by design — exactly three tools (collapsed from a prior set of narrow per-topic tools): query + the schema doc subsume every custom editorial / coverage question, and fewer tools means no permission-prompt sprawl. All three are read-only and return a single JSON object; errors come back as the REST error envelope ({ ok:false, error, code, hint }) so an agent can self-correct on hint. The usual flow is two tools: describe_schema() once, then query(sql) as many times as you like — reach for agentic_query only for genuinely hard, multi-step questions.

toolwhat it doesREST
describe_schema() The DATA MAP — call it FIRST. Returns the full schema doc for the SQL surface: every queryable relation (the curated spine + rollups and the parquet-backed sources — per-day GA4 / Chartbeat, the raw reddit corpus, the FB ad library) with its grain + key columns, the join recipes that bridge them, the safety/limit contract, and copy-pasteable worked SELECTs. GET /v1/query/schema
query(sql) Run ONE sandboxed read-only SELECT over the whole catalog — the universal data path. Compose your own filters / joins / grouping / window. Result is capped at 1000 rows (truncated:true if more exist). POST /v1/query
agentic_query(question) Escape hatch for HARD / multi-step questions: an inner on-box agent EXPLORES the catalog over several rounds (run SQL → read rows → JOIN → refine) then synthesizes a grounded answer. Slower + costlier than query — prefer query + describe_schema for anything you can express directly. none — orchestrates the two above

describe_schema()

Read this first, and again any time you are unsure which relations or columns exist. It returns { markdown, limits } — the full schema documentation plus the caps enforced server-side (max_rows 1000, timeout_sec 300, memory a soft limit that spills to NVMe, max_concurrency, and full_catalog: true meaning the WHOLE catalog is queryable, no relation whitelist).

describe_schema()

query(sql)

The universal data path: compose one read-only SELECT and run it. The contract (enforced server-side; a violation returns code:"rejected", never a crash): a single SELECT or WITH … SELECT — no writes / DDL / DML, no SET/PRAGMA/ATTACH/COPY/CALL, no file functions (read_csv/read_parquet/glob — the predefined views read their fixed parquet for you), no second statement. The whole catalog is readable; output is wrapped in LIMIT 1000 (truncated:true → aggregate or add your own ORDER BY … LIMIT). 300s wall-clock budget, ~20-way concurrency (a 21st gets code:"busy"). For speed, filter on the partition key dt on the date-partitioned sources. Returns { columns, rows, row_count, truncated }.

query("SELECT page_path, title, bp_views FROM v_articles ORDER BY bp_views DESC LIMIT 5")

agentic_query(question)

The escape hatch for questions that are awkward to write as a single SELECT (multi-step, comparative, exploratory). An inner on-box claude (Haiku, from the subscription pool — never the paid API) does NL → SQL → EXPLORE → synthesis over up to ~10 rounds, but runs tool-less: it can only emit SQL text, which the server runs through the same guarded query path — so it can never bypass the safety harness or escape the box. It early-exits as soon as the rows suffice, has a hard ~300s budget, and returns { answer, queries, rounds, elapsed_s, partial, note }. Costs pool tokens per call.

agentic_query("Which feeder subreddits had the worst coverage drop over the last 30 days, and what changed?")

Full field-by-field shapes — including the queryable relations, columns, and join recipes that describe_schema() returns — are in the REST API usage section of the docs (the MCP tools return the same payloads as the REST endpoints listed above).
